The one new finding — billing trigger

NEW  The billing-column protection trigger never recognizes the service role, so it fails closed.

supabase/subscription.sql guards profiles.plan / stripe_* so a user can't self-upgrade from the browser. The guard tests current_user <> 'service_role' — but the function is SECURITY DEFINER, and inside such a function current_user is the function owner (e.g. postgres), never the calling session. So the check is essentially always true.

Effect: it blocks the browser self-upgrade (the intended part) but also blocks the legitimate Stripe-webhook and admin plan update — the service-role client reaches Postgres as role service_role via PostgREST, which the trigger can't see. Paid subscriptions would silently fail to activate. It hasn't bitten yet only because paywalls are off (PAYWALLS_DEFAULT = false), so no real plan change has been attempted.

Fix (in the migration): check the JWT role claim — current_setting('request.jwt.claims', true)::jsonb ->> 'role' — which PostgREST sets per request and SECURITY DEFINER doesn't rewrite. Verify against prod before enabling paywalls: a Stripe test checkout should set plan, and a browser attempt to change it should still raise.

Findings not yet covered by a PR (lower severity)

MED Non-functional PKCE on Twitter/X & TikTok OAuth

lib/clipflow/platforms.ts hardcodes code_challenge: 'challenge' with method plain and sends the same constant as the verifier. That's PKCE in name only. Residual risk is limited — these are confidential server-side clients with a real client_secret, and the state cookie still blocks CSRF — but the PKCE layer does nothing. Fix: generate a random verifier per authorize call, store it in an httpOnly cookie, send S256(verifier) as the challenge.

LOW Encryption-at-rest hardening

Two schemes coexist: ClipFlow tokens use scryptSync with a static salt and fall back to reusing SUPABASE_SERVICE_ROLE_KEY as key material; agent tokens use a bare sha256(APP_ENCRYPTION_KEY). Both are AES-256-GCM with a random IV, so neither is exploitable on its own — but consolidating on one dedicated APP_ENCRYPTION_KEY with a per-value random salt would be cleaner.

LOW User enumeration & raw error messages

auth/signup returns a distinct 409 for an existing email (contrast with auth/resend, which deliberately doesn't). Several 500s return raw err.message, and the Google OAuth callback reflects the token-exchange error into the redirect URL — minor internal info disclosure.
